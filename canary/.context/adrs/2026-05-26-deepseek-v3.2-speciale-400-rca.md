# RCA — `gloo-deepseek-v3.2-speciale` returns HTTP 400 (reproduced; bona-fide bug)

## TL;DR

The 2026-05-25 weekly Full sweep (`canary-probe-j7qvk`) flagged
`v2/model/gloo-deepseek-v3.2-speciale` RED with HTTP 400. Reproduced manually
with `pnpm canary:repro` (replays the exact probe path): the `-speciale` variant
returns **400 deterministically (2/2)** while the base `gloo-deepseek-v3.2`
returns **200 (2/2)** with the byte-identical request body. The fault is
model-specific, not the request → a Gloo↔AtlasCloud provider routing/config
defect for the speciale variant.

## Reproduction

2026-05-26 ~09:42 CT · account `patrickt@servant.io` · canary OAuth client.

```bash
cd canary
pnpm canary:repro v2/model/gloo-deepseek-v3.2-speciale v2/model/gloo-deepseek-v3.2 --repeat 2
```

| Signature | max_tokens | Attempts | Result |
|---|---|---|---|
| `v2/model/gloo-deepseek-v3.2-speciale` | 2048 | 2 | **400 / FAIL ×2** |
| `v2/model/gloo-deepseek-v3.2` (base, control) | 2048 | 2 | 200 / PASS ×2 |

The two requests differ only by `model`. Error envelope: Gloo `INVALID_REQUEST`
(1001, `category: client_error`, `fault: client`, `retryable: false`) wrapping
upstream AtlasCloud `{"code":400,"msg":"bad request"}`, `provider_name:
AtlasCloud`, `is_byok: false`. The `fault: client` label is misleading — the
same body works for the base model, so the request is valid; the fault is in how
Gloo maps/forwards the speciale variant to AtlasCloud.

## Documentation validation

- Live registry `GET /platform/v2/models`: `gloo-deepseek-v3.2-speciale` listed
  as a first-class, priced model — `provider_id: deepseek/deepseek-v3.2-speciale`,
  family `Open Source`, `supports_tools: true`, `max_output_tokens: 163840`,
  no `status`/`beta`/`deprecated` field.
- Mintlify [Supported Models](https://docs.gloo.com/api-guides/supported-models)
  renders that registry live and presents all models uniformly as supported; no
  known-issue concept exists.

Conclusion: the failure is not documented or expected. **Bona-fide bug.**
