# RCA — `gloo-google-gemini-2.5-pro` returns HTTP 503 (reproduced; bona-fide bug, NOT the 2026-04-27 budget class)

## TL;DR

The 2026-05-25 weekly Full sweep (`canary-probe-j7qvk`) flagged
`v2/model/gloo-google-gemini-2.5-pro` RED with HTTP 503
(`PROVIDER_UNAVAILABLE`), its second consecutive weekly failure (also 05-18,
`canary-probe-6w7gc`). Reproduced manually with `pnpm canary:repro`: **7/7
failures across `max_tokens` 2048, 8192, AND server-default (no max_tokens)**, at
~500ms each, while sibling Gemini models pass right now. The server-default
control rules out the 2026-04-27 reasoning-budget false-positive class → this is
a genuine, persistent provider unavailability.

## Reproduction

2026-05-26 ~09:42 CT · account `patrickt@servant.io` · canary OAuth client.

```bash
cd canary
pnpm canary:repro v2/model/gloo-google-gemini-2.5-pro --repeat 3
pnpm canary:repro v2/model/gloo-google-gemini-2.5-pro --max-tokens 8192 --repeat 2
pnpm canary:repro v2/model/gloo-google-gemini-2.5-pro --no-max-tokens --repeat 2
pnpm canary:repro v2/model/gloo-google-gemini-2.5-flash v2/model/gloo-google-gemini-3.1-pro
```

| Signature | max_tokens | Attempts | Result | Latency |
|---|---|---|---|---|
| `gloo-google-gemini-2.5-pro` | 2048 (fixture default) | 3 | **503 / FAIL ×3** | ~510–585ms |
| `gloo-google-gemini-2.5-pro` | 8192 | 2 | **503 / FAIL ×2** | ~485–525ms |
| `gloo-google-gemini-2.5-pro` | none (server default) | 2 | **503 / FAIL ×2** | ~485–510ms |
| `gloo-google-gemini-2.5-flash` (control) | default | 1 | 200 / PASS | 2684ms |
| `gloo-google-gemini-3.1-pro` (control) | default | 1 | 200 / PASS | 9112ms |

Envelope: `service_unavailable_error` (2002, `PROVIDER_UNAVAILABLE`,
`fault: provider`, `retryable: true`). 7/7 manual failures plus the two weekly
canary runs (05-18, 05-25) ≈ 9 consecutive failures over a week.

## Why this is NOT a repeat of the 2026-04-27 reasoning-budget false positive

The prior RCA (`2026-04-27-reasoning-model-max-tokens-rca.md`) found 2.5-pro
503s were self-inflicted: `max_tokens=48` exhausted the thinking budget, the
platform converted the empty completion into a 503, and three bug reports were
filed before a Gloo engineer caught it. The decisive distinguisher then: casual
usage with **no** `max_tokens` could not reproduce it.

This time the `--no-max-tokens` control **does** reproduce the 503 (2/2), at the
same fast ~500ms latency as the capped runs, while sibling Gemini models
succeed. That is the opposite signature: the failure is independent of
`max_tokens`, so it is a real provider-side unavailability, not a budget
artifact. The registry record also shows `max_output_tokens: 65500` and
`supports_reasoning: false`, further ruling out a token-range / reasoning cause.

## Documentation validation

- Live registry `GET /platform/v2/models`: `gloo-google-gemini-2.5-pro` listed
  as first-class and priced — `max_output_tokens: 65500`,
  `supports_reasoning: false`, no `status`/`beta`/`deprecated` field.
- Mintlify [Supported Models](https://docs.gloo.com/api-guides/supported-models):
  presents all models uniformly as supported; no known-issue concept exists.

Conclusion: the failure is not documented or expected. **Bona-fide bug.**

## Classification

| Verdict | Root-cause owner | Bug? |
|---|---|---|
| 503 PROVIDER_UNAVAILABLE | Gloo provider availability (2.5-pro backend) | **Yes — file** |

Not a canary bug, and (this time) not a canary-request artifact. Upgraded from
the triage's initial "monitor/flag" to a real report on the strength of the
no-max-tokens control + healthy-sibling control + the 9-failure history.

## Recommended actions

1. File a Gloo bug report citing the no-max-tokens control (rules out the
   2026-04-27 class), the healthy-sibling control, and the consecutive-failure
   history.
2. Canary hardening (separate, lower priority): add a single bounded retry on
   `retryable: true` 5xx before declaring RED, in
   `canary/src/probes/v2-completions.ts`. NOTE this would NOT have suppressed
   this RED (every attempt fails) — which is the point: a persistent RED past
   one retry is real signal.
